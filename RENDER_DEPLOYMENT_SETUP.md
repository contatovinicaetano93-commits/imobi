## PostgreSQL Setup

### Step 1: Create PostgreSQL Database on Render

1. Go to [Render Dashboard](https://dashboard.render.com)
2. Click **"New +"** → **"PostgreSQL"**
3. Configure the instance:
   - **Name**: `imobi-staging-db`
   - **Database Name**: `imobi_staging`
   - **Region**: Select region matching your deployment (e.g., `Ohio` for US-East)
   - **PostgreSQL Version**: `14.9` (minimum 14+)
   - **Instance Type**: `Standard` (starter can work, upgrade if needed)
   - **Pricing**: Auto-sleep enabled recommended for staging

4. Click **"Create Database"**
5. Wait for database to be ready (typically 2-3 minutes)

### Step 2: Retrieve PostgreSQL Connection String

Once the database is ready:

1. Click on the database instance
2. In the **"Connections"** section, copy the **"Internal Database URL"** (if deploying from Render) or **"External Database URL"** (if deploying from outside Render)
3. Format: `postgresql://username:password@host:5432/imobi_staging`

**Example:**
```
postgresql://imbobi:A8x9Kp2mL5Q3vN1x@dpg-abc123.postgres.render.com:5432/imobi_staging
```

### Step 3: Create a Dedicated Database User (Optional but Recommended)

For enhanced security, create a staging-specific user instead of using the admin user:

1. Connect to the database using Render's built-in query editor or `psql`:
```bash
psql postgresql://username:password@host:5432/imobi_staging
```

2. Create a staging user:
```sql
CREATE USER imbobi_staging WITH PASSWORD 'your-secure-password-here';
GRANT CONNECT ON DATABASE imobi_staging TO imbobi_staging;
GRANT CREATE ON SCHEMA public TO imbobi_staging;
```

3. Use this user's connection string instead:
```
postgresql://imbobi_staging:your-secure-password@dpg-abc123.postgres.render.com:5432/imobi_staging
```

## Redis Cache Setup

### Step 1: Create Redis Instance on Render

1. Go to [Render Dashboard](https://dashboard.render.com)
2. Click **"New +"** → **"Redis"**
3. Configure the instance:
   - **Name**: `imobi-staging-cache`
   - **Region**: Same region as PostgreSQL
   - **Redis Version**: `7.x` (minimum 7+)
   - **Instance Type**: `Standard` (starter can work, upgrade if needed)
   - **Eviction Policy**: `allkeys-lru` (recommended for cache)

4. Click **"Create Redis"**
5. Wait for instance to be ready (typically 1-2 minutes)

### Step 2: Retrieve Redis Connection String

Once the Redis instance is ready:

1. Click on the Redis instance
2. In the **"Connections"** section, copy the **"Internal Redis URL"** (if deploying from Render) or **"External Redis URL"** (if deploying from outside)
3. Format: `redis://:password@host:port`

**Example:**
```
redis://:cA3xL9pK2mQ5vN1x8@dpg-def456.redis.render.com:6379
```

### Step 3: Configure Redis Password

The password is auto-generated by Render. To ensure it's strong:

1. If you need to reset it:
   - Click on the Redis instance
   - **"Settings"** → **"Change Password"**
   - Render will generate a new secure password

## Database Migration

### Prerequisites

Ensure you have:
- Access to the imobi repository locally or in your deployment environment
- `pnpm` installed
- `DATABASE_URL` environment variable set

### Step 1: Run Prisma Migrations

From the repository root, run:

```bash
# Set the connection string
export DATABASE_URL="postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging"

# Generate Prisma client
pnpm db:generate

# Run migrations (creates all tables, indexes, enums)
pnpm db:migrate
```

### Step 2: Verify Migration Success

Check that all tables were created:

```bash
export DATABASE_URL="postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging"

# Open Prisma Studio for visual inspection
pnpm prisma:studio
```

This opens an interface at `http://localhost:5555` where you can inspect all tables:
- `Usuario` - User accounts
- `Credito` - Credit lines
- `Obra` - Construction projects
- `EtapaObra` - Project stages
- `EvidenciaEtapa` - Photo evidence
- `LiberacaoParcela` - Payment releases
- `KycDocumento` - KYC documents
- `Notificacao` - Push notifications
- `ScoreHistorico` - User score history
- `JobFalha` - Failed job queue entries
- `AnalyticsEvent` - Event tracking

### Step 3: Optional - Seed Initial Data

To populate staging with test data:

```bash
export DATABASE_URL="postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging"

# Run seed script (located at services/api/src/seeds/seed.ts)
pnpm --filter @imbobi/api seed
```

**Note**: Seed script is optional. Check `/services/api/src/seeds/README.md` for details.

### Generating Secrets

Generate strong secrets using Node.js:

```bash
# JWT_SECRET (base64, minimum 64 characters)
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
# Output: 2vL8mK5xQ1pN9aBc7dEfGhIjKlMnOpQrStUvWxYzAbCdEfGhIjKlMnOp

# ENCRYPTION_KEY (base64, 32 bytes)
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
# Output: aBcDeFgHiJkLmNoPqRsTuVwXyZaBcDeFgHiJkLmNo
```

Store these **ONLY** in Render environment variables. Never commit to git.

## Security Best Practices

### 1. Database Security

**Do:**
- Use strong passwords (32+ characters, mix of upper/lower/numbers/symbols)
- Create dedicated staging user (don't use admin/root user)
- Enable SSL/TLS for connections (enabled by default on Render)
- Restrict network access via firewall rules if available

**Don't:**
- Store credentials in git or `.env` files
- Use simple passwords like `password123`
- Expose DATABASE_URL in logs or error messages
- Use the same credentials across environments (dev/staging/prod)

### 2. Redis Security

**Do:**
- Use strong passwords auto-generated by Render
- Keep Redis URL private
- Monitor memory usage

**Don't:**
- Disable authentication
- Expose Redis to public internet
- Use same password across environments

### 3. API Secret Management

**Do:**
- Rotate `JWT_SECRET` annually or on compromise
- Store all secrets in Render dashboard environment variables
- Use different secrets per environment
- Log secret rotation events

**Don't:**
- Hardcode secrets in code
- Include secrets in Docker images
- Share secrets via email or chat

### 4. Network Isolation

For maximum security, use Render's Private Services (if available):
- Database: Set to private network only
- Redis: Set to private network only
- API: Can be public, but only accepts requests from authorized sources

## Monitoring & Alerts

### PostgreSQL Monitoring

**In Render Dashboard:**

1. Click on PostgreSQL instance
2. Go to **"Monitoring"** tab
3. Monitor these metrics:
   - **Connections**: Should be < 20 for staging
   - **Database Size**: Alert if exceeds 5GB
   - **CPU Usage**: Alert if exceeds 80%
   - **Memory Usage**: Alert if exceeds 80%

**Set up alerts:**
- Click **"Monitoring"** → **"Alert Rules"**
- Create alerts for:
  - Connections > 30
  - CPU > 80% for 5 minutes
  - Database unreachable

### Redis Monitoring

**In Render Dashboard:**

1. Click on Redis instance
2. Go to **"Monitoring"** tab
3. Monitor these metrics:
   - **Memory Usage**: Should be < 80% of total
   - **Evicted Keys**: If non-zero, memory needs increase
   - **Connected Clients**: Should be < 100 for staging

**Set up alerts:**
- Memory usage > 75%
- Evicted keys > 100/min
- Redis unreachable

### API Health Checks

Configure Render service health checks:

1. Go to API service settings
2. **"Health Checks"** section
3. Set:
   - **Endpoint**: `/health`
   - **Interval**: 30 seconds
   - **Timeout**: 5 seconds
   - **Failure threshold**: 3 failures

### Log Monitoring

**Search for connection errors:**
```bash
# From API service logs, look for:
"Database connection failed"
"Redis connection failed"
"Connection pool exhausted"
```

## Backup & Recovery

### Automated Backups

**PostgreSQL Backups (Render):**

1. PostgreSQL instance → **"Settings"**
2. **"Backups"** section
3. Backups are automatically created daily (retention: 7 days for Standard tier)
4. To enable point-in-time recovery (if available):
   - Contact Render support or check dashboard

**Backup Policy:**
- Daily automated backups
- Retention: 7 days minimum
- Test restore procedure monthly

### Manual Backup (Staging)

If you need a backup before migrations:

```bash
# Using pg_dump
pg_dump postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging > imobi_staging_backup_$(date +%Y%m%d).sql

# Compressed (recommended)
pg_dump postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging | gzip > imobi_staging_backup_$(date +%Y%m%d).sql.gz
```

### Restore from Backup

```bash
# Restore from file
psql postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging < imobi_staging_backup_20260602.sql

# Or (compressed):
gunzip < imobi_staging_backup_20260602.sql.gz | psql postgresql://imbobi_staging:password@dpg-abc123.postgres.render.com:5432/imobi_staging
```

### Recovery Checklist

After restoring from backup:

- [ ] Verify all tables exist: `\dt` in psql
- [ ] Check record counts
- [ ] Test API connection
- [ ] Verify no data inconsistencies
- [ ] Check application logs for errors
- [ ] Run health checks

### Disaster Recovery Plan

For staging environment loss:

1. **Drop corrupted database** (if necessary):
```bash
# WARNING: This deletes all data
DROP DATABASE imobi_staging;
CREATE DATABASE imobi_staging;
```

2. **Restore from backup or run migrations**:
```bash
DATABASE_URL="..." pnpm db:migrate
```

3. **Re-seed data** (if applicable):
```bash
DATABASE_URL="..." pnpm --filter @imbobi/api seed
```

4. **Verify data integrity**:
```bash
# Count records in key tables
SELECT COUNT(*) FROM "Usuario";
SELECT COUNT(*) FROM "Credito";
SELECT COUNT(*) FROM "Obra";
```

## Support & Troubleshooting

### Common Issues

**Connection Refused**
- Verify DATABASE_URL is correct (copy from Render dashboard)
- Check if PostgreSQL instance is in "Available" state
- Verify firewall allows connections (usually automatic on Render)

**Redis Connection Failed**
- Verify REDIS_HOST, REDIS_PORT, REDIS_PASSWORD
- Check Redis instance is in "Available" state
- Look for connection timeout errors in API logs

**Migration Errors**
- Ensure DATABASE_URL is set before running `pnpm db:migrate`
- Check that database user has CREATE privilege on schema
- Review error message for specific SQL issues

**Out of Memory**
- For Redis: Increase instance size in Render dashboard
- For PostgreSQL: Check query optimization or increase instance

# Check all environment variables in API service
