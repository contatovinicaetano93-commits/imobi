SYSTEM ARCHITECTURE
================================================================================

Current Email Flow:
-------------------

User Action (e.g., registration)
  ↓
API Service (e.g., /auth/registrar)
  ├→ Process request (create user in DB)
  └→ Call EmailService async (fire-and-forget)
       ├→ Detection: Check EMAIL_PROVIDER env var
       └→ Send via selected provider
            ├→ SendGrid SMTP (production) ← CURRENTLY DISABLED
            ├→ AWS SES (alternative)
            ├→ Generic SMTP (fallback)
            └→ Console logging (development)
  ↓
Return response to user (IMMEDIATE - doesn't wait for email)
  ↓
Email delivered async (2-5 seconds later, independent of request)

Key Point: Email sending doesn't block user workflows!


PROVIDER SELECTION
================================================================================

When application starts, it checks EMAIL_PROVIDER env var:

   if EMAIL_PROVIDER = "sendgrid":
       ├─ Read SENDGRID_API_KEY
       ├─ Connect to smtp.sendgrid.net:587 (TLS)
       ├─ Authenticate with user="apikey", pass=API_KEY
       └─ Ready to send

   else if EMAIL_PROVIDER = "ses":
       ├─ Read AWS credentials
       ├─ Connect to AWS SES SMTP
       └─ Ready to send

   else if EMAIL_PROVIDER = "smtp":
       ├─ Read SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS
       ├─ Connect to generic SMTP server
       └─ Ready to send

   else (default):
       ├─ No transporter initialized
       └─ Fall back to console logging (development mode)


INTEGRATION POINTS DETAIL
================================================================================

1. REGISTRATION WORKFLOW
   ─────────────────────
   Endpoint: POST /auth/registrar
   File: services/api/src/modules/auth/auth.service.ts

   Flow:
     1. Validate input (email, password, name, etc.)
     2. Hash password with bcrypt
     3. Create user in database
     4. Generate JWT tokens
     5. Call: emailService.bemVindoEmail(nome, email)
     6. Return JWT to client (IMMEDIATE)
     7. Email arrives async in inbox (2-5 seconds later)

   Template: Welcome email with next steps
   Impact: If email fails, registration still succeeds
   Error Handling: Logged but doesn't block


2. KYC APPROVAL WORKFLOW
   ─────────────────────
   Endpoint: PATCH /kyc/{id}/aprovar
   File: services/api/src/modules/kyc/kyc.service.ts

   Flow:
     1. Find KYC document by ID
     2. Update status to "APROVADO"
     3. Call: emailService.kycAprovadoEmail(nome, email)
     4. Check if KYC complete (all docs approved)
     5. If complete: Update user kycStatus
     6. Return success (IMMEDIATE)
     7. Email arrives async

   Template: KYC approval confirmation
   Impact: If email fails, approval still succeeds
   Error Handling: Logged but doesn't block


3. KYC REJECTION WORKFLOW
   ──────────────────────
   Endpoint: PATCH /kyc/{id}/rejeitar
   File: services/api/src/modules/kyc/kyc.service.ts

   Flow:
     1. Find KYC document
     2. Update status to "REJEITADO"
     3. Call: emailService.kycRejeitadoEmail(nome, email, motivo)
     4. Return success (IMMEDIATE)
     5. Email arrives async with retry link

   Template: Rejection with reason and "try again" link
   Impact: If email fails, rejection still succeeds
   Error Handling: Logged but doesn't block


4. STAGE APPROVAL WORKFLOW
   ────────────────────────
   Endpoint: PATCH /etapas/{id}/aprovar
   File: services/api/src/modules/etapas/etapas.service.ts

   Flow:
     1. Find stage (etapa) by ID
     2. Verify obra is active
     3. Verify at least one evidence photo exists
     4. Update stage status to "CONCLUIDA"
     5. Create LiberacaoParcela record (payment entry)
     6. Build WhatsApp message for finance team
     7. Call: emailService.capitalFaseAguardandoPagamentoEmail(...)
     8. Create in-app notification
     9. Send push notification
    10. Return success (IMMEDIATE)
    11. Email arrives async with WhatsApp link

   Template: Stage approval with capital amount and WhatsApp link
   Integration: Includes WhatsApp to contact finance team
   Impact: Multiple notification channels (email + in-app + push)
   Error Handling: Email failure doesn't block workflow


5. INSTALLMENT RELEASE WORKFLOW (ASYNC)
   ───────────────────────────────────
   File: services/api/src/workers/liberacao-parcela.worker.ts
   Type: BullMQ background job (runs hourly)

   Flow:
     1. Find all "PENDENTE" (pending) payment releases
     2. For each release:
        a. Lock it (idempotency guard)
        b. Transfer funds (virtual, updates DB)
        c. Update status to "CONCLUIDA"
        d. Call: emailService.parcelaLiberadaEmail(...)
        e. Create in-app notification
        f. Send push notification
     3. Log completion
     4. Next run: 1 hour later

   Template: Payment confirmation with amount
   Trigger: Hourly background job (independent of user requests)
   Impact: Multiple notification channels (email + in-app + push)
   Error Handling: Email failure logged, but doesn't block payment


6. PASSWORD RECOVERY WORKFLOW
   ──────────────────────────
   Endpoint: POST /auth/esqueceu-senha
   File: services/api/src/modules/auth/auth.service.ts

   Flow:
     1. Find user by email
     2. Generate reset token (32-byte random)
     3. Store token in DB with 1-hour expiry
     4. Call: emailService.recuperacaoSenhaEmail(nome, email, token)
     5. Return success message (IMMEDIATE)
     6. Email arrives async with reset link

   Template: Password reset with token link (1-hour expiry)
   Impact: If email fails, user can retry request
   Error Handling: Logged but doesn't block token generation

RISK ASSESSMENT
================================================================================

Very Low Risk Deployment

Potential Issues:
  1. SendGrid API key invalid
     - Impact: Emails not sent (falls back to console)
     - Detection: Immediate in logs
     - Recovery: Regenerate key, redeploy (5 min)

  2. Sender email not verified
     - Impact: Emails bounced
     - Detection: SendGrid dashboard
     - Recovery: Verify sender, retry (24 hours)

  3. Email service down
     - Impact: Emails queued, then retried
     - Detection: SendGrid dashboard
     - Recovery: Automatic when service recovers

Mitigation:
  • Test before going live (5 min)
  • Monitor first week daily
  • Can rollback in 5 minutes (set EMAIL_PROVIDER=console)
  • No user-facing risk
